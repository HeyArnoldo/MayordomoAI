# Delta for Agent Chat

## ADDED Requirements

### Requirement: Image Parts in Agent Turn

The agent service MUST accept message payloads that include one or more image parts (base64 data-URLs) in addition to the optional text part. These parts MUST be passed to `streamText` / `convertToModelMessages` as first-class image content blocks compatible with gpt-4o vision. The agent MUST respond in the conversation locale regardless of image content language.

#### Scenario: Agent receives a receipt image

- GIVEN a message payload containing one image part (a receipt photo) and no text part
- WHEN the agent processes the turn
- THEN `streamText` receives the image as an image content block
- AND the model produces a reply extracting amount, merchant, and date as a suggestion
- AND the reply is in the conversation locale

#### Scenario: Agent receives image plus text

- GIVEN a message payload containing one image part and a text part ("What is the total?")
- WHEN the agent processes the turn
- THEN both the image and the text are passed to the model in the same turn
- AND the model reply addresses both the image content and the user's question

### Requirement: mediaContext Metadata Persistence

The `Message` entity MUST include a nullable `mediaContext` JSONB column. When a message carries image attachments, the controller MUST persist `mediaContext` with the following shape and MUST NOT persist the image binary.

```
mediaContext: {
  type: "image",
  files: [
    {
      filename: string,   // original filename or WhatsApp media ID
      size: number,       // bytes
      mime: string        // e.g. "image/jpeg"
    }
  ]
}
```

`mediaContext` MUST be `null` for text-only messages (backward-compatible default).

#### Scenario: Image message persisted

- GIVEN a message with one image attachment is processed successfully
- WHEN the message is saved to the database
- THEN `mediaContext` contains `type: "image"` and one entry in `files` with filename, size, and mime
- AND no image binary data appears anywhere in the `Message` row

#### Scenario: Text-only message persisted

- GIVEN a message with no attachments
- WHEN the message is saved to the database
- THEN `mediaContext` is `null`
- AND the rest of the message fields are unchanged (backward compatibility)

#### Scenario: Multiple images persisted

- GIVEN a message with two image attachments
- WHEN the message is saved to the database
- THEN `mediaContext.files` has exactly two entries, one per image

### Requirement: messageSchema Contract Extension

The shared `messageSchema` (in `packages/contracts`) MUST be extended with an optional `mediaContext` field matching the shape defined in the `mediaContext Metadata Persistence` requirement. Existing text-only message payloads that omit `mediaContext` MUST continue to pass schema validation without modification.

#### Scenario: Schema accepts image message

- GIVEN a message object with `mediaContext: { type: "image", files: [...] }`
- WHEN the object is validated against `messageSchema`
- THEN validation passes

#### Scenario: Schema accepts text-only message (backward compatibility)

- GIVEN a message object with no `mediaContext` field
- WHEN the object is validated against `messageSchema`
- THEN validation passes without errors

#### Scenario: Schema rejects invalid mediaContext shape

- GIVEN a message object where `mediaContext.files` is missing or not an array
- WHEN the object is validated against `messageSchema`
- THEN validation fails with a descriptive error

---

### Requirement: Vision Usage Tracking

AI token usage generated by vision-capable model calls MUST be recorded under the existing `ai-usage` tracking mechanism with `kind: 'agent'`. No new tracking kind is introduced.

#### Scenario: Vision call usage recorded

- GIVEN the agent processes a message containing an image part
- WHEN the model responds and usage data is available
- THEN a usage record is written with `kind: 'agent'` including prompt and completion token counts
- AND the record is indistinguishable in structure from a text-only agent usage record

---

### Requirement: Receipt Prompt Hint

The agent system prompt MUST include a receipt-analysis hint that instructs the model to extract amount, merchant, and date from receipt images and present them as a suggestion for the user to confirm before any financial tool is invoked. No new `analyzeReceipt` tool is added.

#### Scenario: Agent replies to receipt with structured suggestion

- GIVEN the system prompt includes the receipt hint
- WHEN the user sends a receipt image
- THEN the agent reply includes amount, merchant, and date extracted from the image
- AND the reply presents these as a suggestion pending user confirmation
- AND no transaction is registered automatically
